Day-to-day testing is still just `npx expo start` from here on; only native dependency changes need a rebuild
- `src/lib/storage.ts`: MMKV instance (`createMMKV({ id: 'stocklog' })`), with an in-memory stub on web — MMKV's own web fallback (`localStorage`-backed) throws outside a DOM, which crashes the web SSR bundle exactly like the SecureStore issue from Stage 2
- `src/lib/offline-context.tsx` (`OfflineProvider`/`useOffline`): the single source of truth for `inventory` and `sales` data app-wide, replacing each screen's own direct Supabase fetch
  - Caches the last-fetched `inventory`/`sales` to MMKV and restores from it synchronously on launch, so Stock/Sales/Ledger show real data immediately even if the app opens offline
  - `addInventory`/`recordSale`: try the real Supabase call first when online (so duplicate-IMEI and other validation errors still surface immediately, same as before); only fall back to queueing when the call fails with a connectivity-shaped error (no Postgres error `.code`, i.e. the request never reached the server) or when `NetInfo` already reports offline
  - Queued operations use a client-generated UUID (`expo-crypto`'s `randomUUID()`) as the row's real `id` so dependent actions work immediately — e.g. you can sell a device in the same offline session it was added in, before either has touched the server
  - `inventory`/`sales` are pure derived views: cached server rows merged with an optimistic overlay computed from the pending outbox (pending adds inserted, sold items flagged, pending sales synthesized using the already-merged inventory for device name lookup) — screens never see "raw cache" vs "raw pending" as separate concepts
  - Sync queue replays in order on reconnect, stopping at the first failure (so a sale that depends on an earlier queued add never jumps ahead of it); a genuine IMEI conflict discovered at sync time surfaces as a banner message instead of vanishing silently
- Added an offline/pending-sync banner to `(tabs)/_layout.tsx` (amber while offline or syncing, clay on a real sync error), shown above whichever tab is active
- `add.tsx` and `sell/[id].tsx` now call `addInventory`/`recordSale` from the provider instead of talking to Supabase directly; `sell/[id].tsx` looks up the device from the already-merged `inventory` list instead of fetching it by id (which would fail for a device added offline, since it wouldn't exist server-side yet)
- Bug found and fixed during the native build: CocoaPods' Ruby internals crash (`Encoding::CompatibilityError`) on this machine because the shell doesn't export a UTF-8 locale. `pod install` must be run with `LANG=en_US.UTF-8 LC_ALL=en_US.UTF-8` — hit this twice (once during my own build, once during the user's terminal re-run) before it was clear it's a standing environment quirk, not a one-off

- **Fixed: a deleted item kept showing in the Stock list.** Root cause: `deleteInventory`'s online-success path only called `void refresh()` (fire-and-forget) and relied on the refetch round-trip to update the UI — same pattern as `addInventory`/`updateInventory`, but for a delete the lingering stale item reads as "it didn't work" rather than just slightly-stale data. **Fourth follow-up — found the real cause of "list starts from the middle" (it wasn't scroll position) and the period selector crowding its neighbor:**
- Both reports traced to the **same root cause**: `stock.tsx`'s `columnHeader` had `marginTop: -Spacing.two` / `marginBottom: -Spacing.two` (added two rounds ago to visually tighten the gap around it) sitting inside a parent `View` that also used the flexbox `gap` property. Negative margins combined with `gap` apparently don't reduce the effective spacing the way they would with a plain margin-only layout — Yoga (RN's layout engine) computed a *much larger* gap instead, most visible/obvious once a brand filter narrowed the list down to one item (a huge blank area appeared between the filter chips and the column header/row, which read as "the list starts from the middle of the screen" even though no scrolling was involved at all). Fixed by removing `gap` from Stock's `safeArea` entirely and switching to explicit positive `marginBottom` on each section (`headerRow`, `search`, a new wrapper around `FilterTabs`), matching the margin-only spacing pattern Ledger/Sales already used safely — no more `gap`+negative-margin combination anywhere in the app now
- The period selector "interfering with what's underneath it" on Ledger/Sales was a separate, simpler bug: the new `<SegmentedControl>` had been dropped in with no margin at all, sitting flush against the stat trio below it. Fixed by wrapping it in a `periodWrap` View with `marginBottom: Spacing.three` on both screens
- **Lesson:** negative margins are a fragile way to "pull two elements closer" in a flex layout that also uses `gap` — prefer removing/reducing the `gap` (or switching that section to explicit positive margins) over fighting it with a negative offset, since the interaction isn't reliably predictable across Yoga versions/configurations
- **Added weekly/monthly views to Ledger and Sales.** Both screens now have a 3-way `SegmentedControl` (Today/Week/Month) above the stat trio. New `getPeriodStartDateString`/`periodLabel` helpers in `date.ts` use *trailing* windows (last 7/30 days inclusive of today) rather than calendar week/month boundaries — deliberate choice, see Decisions Log. Ledger's "Recent activity" feed and Sales' day-grouped history list are both re-filtered to the selected period, not just the stat trio. "Stock value" and "Units" on Ledger don't change with the period since they're current snapshots, not historical aggregates
- **Added the ability to restore a sold item to stock** (customer return). Sold rows in Stock are tappable again (previously disabled) and open a one-action sheet, `Restore to stock`. New `restore_sale(p_inventory_id)` Postgres RPC (`supabase/migrations/0004_restore_sale.sql`, same `security invoker` pattern as `record_sale`) deletes the `sales` row and flips `inventory.status` back to `in_stock` atomically; the migration also adds a DELETE RLS policy for `sales` (none existed). `offline-context.tsx` got a new `restoreSale` following the same online-first/offline-queue/optimistic-local-update shape as the other mutations — a queued `restore_sale` op suppresses the matching sale from the merged sales list and forces the inventory item back to `in_stock` locally, even before it syncs
- **Two migrations still need to be run by the user** in the Supabase SQL editor before this session's features fully work: `0003_inventory_delete.sql` (delete) and `0004_restore_sale.sql` (restore). Neither could be applied from this environment — no service-role key or Supabase CLI session available here. Until they're run, delete/restore will fail with an RLS/permission error surfaced through the normal error-alert path (not a silent failure)

**Fifth follow-up — IMEI now optional, Brand/Color now required:**
- `inventory.imei` changed from `unique not null` to `unique` (nullable) — new migration `0005_imei_optional.sql` (`alter table inventory alter column imei drop not null`). The unique constraint is unaffected: Postgres treats multiple `NULL`s as not equal to each other under a unique constraint, so any number of no-IMEI devices (accessories, etc.) can coexist
- `Inventory['imei']`/`AddInventoryPayload['imei']`/`UpdateInventoryPayload['imei']` all changed to `string | null`; `maskImei()` now takes and returns `string | null` (null in, null out) instead of assuming a value always exists. Every call site updated to show "No IMEI" instead of either crashing or interpolating the literal string "null" into a meta line — caught two of these (Ledger's activity feed, the Sell screen's context row) only because `tsc`/manual review caught the type change, not because either would have thrown — `null` renders as nothing in JSX, so a naive search wouldn't have surfaced them

- **App PIN added.** Off by default; turned on from More → Security → App PIN. New `src/lib/pin-context.tsx` (`PinProvider`/`usePin()`) hashes the 4-digit PIN with `expo-crypto` (SHA-256) and stores only the hash in `expo-secure-store` — never the plaintext PIN, same storage mechanism already used for Supabase auth tokens. Re-locks on every background→foreground app-state transition, not just cold launch — locking only at launch would make the PIN trivially bypassable by just suspending the app instead of closing it
- **Sign-up/login basics fixed:** sign-up now has a "Confirm password" field (a typo before meant a locked-out account with no recourse) and a client-side minimum-length check ahead of Supabase's own; both password fields on sign-up/login got a show/hide eye-icon toggle (`PasswordVisibilityToggle`, reusing `TextField`'s existing `labelRight` slot — no change to `TextField` itself); sign-in's error message used to always say "email and password don't match" regardless of the actual error — now passes through the real Supabase error message and only humanizes the specific "Invalid login credentials" case
- **Forgot/reset password flow added, then revised once the user actually hit it on a real device.** First version: `/forgot-password` requested a reset, `/reset-password` (a deep-link completion screen) parsed the recovery link's tokens by hand and exchanged them for a session. The user tried it and got a blank page after tapping the email link. Root cause: Supabase's reset email links to *their own* server first, which then 302-redirects to the app's custom scheme — and modern mobile browsers (Chrome on Android in particular) block automatic redirects to a custom scheme that weren't triggered by a real tap, so the handoff to the app never happens and the browser just shows blank. This isn't fixable from the app side without a hosted web page to bounce through first, which this project doesn't have (mobile-only, no website)
- **Revised to a code-based flow instead of a link**, which Supabase supports natively and sidesteps the redirect problem entirely (nothing to block — no link, no browser hop). `/forgot-password` is now a single screen with two stages: stage 1 emails a 6-digit code (`resetPasswordForEmail`, no `redirectTo` needed anymore); stage 2 takes the code + new password, calls `verifyOtp({ token, type: 'recovery' })` to establish the recovery session, then `updateUser({ password })`. The old `/reset-password` deep-link screen was deleted outright rather than left as a dead/unused fallback
- **Needs one manual step from the user:** edit the "Reset Password" email template in Supabase Dashboard → Authentication → Email Templates to actually include `{{ .Token }}` in the body — the default template only shows the confirmation link, not the code
- Moved `forgot-password.tsx` from inside `(auth)/` to a top-level route, registered unguarded in `_layout.tsx` — caught this before it shipped, not after: `(auth)` is gated by `guard={!session}`, and `verifyOtp` establishes a session *during* stage 2, before `updateUser` runs. Left inside `(auth)`, that session change would flip the guard and risk the screen getting redirected away mid-flow (onto `(tabs)`, since an existing user already has a shop) before the password actually updates. Moving it out removes the ambiguity entirely rather than relying on async-closures-surviving-unmount behavior that's true in practice but not the kind of thing worth depending on
- A recovery session deliberately bypasses the App PIN gate (checked via `usePathname() !== '/forgot-password'` in `_layout.tsx`) — otherwise someone resetting a forgotten password could get stuck behind a PIN they also don't remember
- **Visual polish:** added a small `BrandMark` component (rounded accent square + the splash/icon's glyph) to the top of sign-up, login, forgot-password, and shop-setup — these were plain text-on-paper before, with no visual tie-in to the app's actual identity
- Could not visually verify sign-up/login/forgot-password on the simulator this round — reaching a logged-out state requires tapping through the "Sign out?" confirmation `Alert.alert`, and this environment has no tap automation. The PIN feature itself (toggle, set-pin screen, lock bypass for normal use) was verified via simulator screenshots; the auth screens rely entirely on already-proven shared components (`TextField`, `PrimaryButton`, `ThemedText`/`ThemedView`), so risk is low, but worth the user's own look. The code-based reset flow specifically still needs a real end-to-end test (request → real email → enter code → new password actually works) since it was caught and fixed from a single user report, not a live round-trip in this environment

Any new money display in later stages should reuse `Money`, not reintroduce the bug
- A bug "fixed" while testing in Expo Go isn't necessarily fixed on a real native build — Expo Go and the custom dev client can resolve custom-font glyph fallback differently for the same font files (this is exactly what happened with ₦ in Space Grotesk). Anything font/glyph-related needs reverification on the dev client, not just Expo Go
- Project now requires a custom dev client (`npx expo run:ios`), not plain Expo Go, because of `react-native-mmkv` — this is permanent from Stage 6 onward, not a one-time detour

- Online writes still try the direct Supabase call first and only fall back to the offline queue on a connectivity-shaped failure — chosen so the common (online) case keeps today's immediate validation errors (duplicate IMEI, etc.) instead of becoming eventually-consistent for everyone
- Offline-queued operations sync strictly in FIFO order and stop at the first failure, rather than skipping and continuing — a queued sale can depend on a queued add that hasn't synced yet, so reordering or partial-skipping could violate that dependency
- The StockLog_Ledger.jsx mockup (the file DESIGN.md cited but that had never actually been committed) is now saved at the project root — it's the actual design source of truth, ahead of DESIGN.md's prose. If a future change seems to contradict DESIGN.md but matches the mockup's actual behavior, the mockup wins
- Deep-linking via `xcrun simctl openurl booted "stocklog:///<route>"` only skips the "Open in stocklog?" confirmation dialog when the dev client is already running and foregrounded — useful for screenshotting specific screens without tap automation, but the first link of a session (or one fired while the app is cold) will still need a manual tap to dismiss that dialog
- `ios/` and `android/` are gitignored (standard Expo managed workflow) and are fully disposable — any change to `app.json`'s native-level config (icons, splash, package name) requires `npx expo prebuild --clean` to actually take effect; `expo run:ios`/`expo run:android` alone just rebuild whatever's already in that directory and will silently keep stale native config otherwise

- When a design handoff conflicts with this file's own CLAUDE.md scope rules (e.g. a mockup showing a feature explicitly listed as out-of-scope for MVP), CLAUDE.md wins — visual fidelity to a handoff doesn't override the project's own stated scope boundaries

- The Stock screen's row tap is a fork (sell vs edit), not a default action — using a native `Alert.alert` action sheet rather than introducing a bottom-sheet library or a long-press gesture, since the app already has an established, working pattern for "ask before doing something" (sign-out, now delete) and a third use of the same pattern needs no new dependency or interaction paradigm
- Deleting a device that's still only a local pending add (never synced) is resolved entirely client-side by dropping the outbox op, not by queuing a "delete" that would otherwise race against an "add" that hasn't landed yet — simpler and avoids a whole class of ordering bugs (queuing a delete for a not-yet-existent server row)
- "Week"/"Month" views use trailing windows (last 7/30 days) rather than calendar boundaries — a calendar week/month would show misleadingly small numbers for the first few days of a new week/month, which is a worse default for a dealer just glancing at "how am I doing" than a rolling window is
- Restoring a sold item deletes its `sales` row rather than soft-flagging it as "returned" — the ask was specifically "restore... incase customer returns it" (undo), not "track returns as their own category of event"; revisit only if asked for return analytics specifically
- Avoid mixing flexbox `gap` with negative margins on the same axis anywhere in this codebase — confirmed unpredictable in practice (see the Stock screen "list starts from the middle" bug). When a section needs tighter spacing than its siblings, prefer removing that container's `gap` in favor of explicit per-child margins instead of trying to negative-margin a gap-managed layout
- The App PIN is local-device-only, never synced to Supabase — it's a "someone picked up the unlocked phone" deterrent, not an account credential, so it doesn't need to follow the user across devices or survive a reinstall. Storing only a SHA-256 hash (never the plaintext PIN) in SecureStore was a deliberate defense-in-depth choice even though SecureStore is already Keychain/Keystore-encrypted at rest
- The PIN re-locks on every background→foreground transition rather than only on cold launch — a security feature that only checks once per app-process-lifetime is trivially defeated by switching apps instead of fully closing one, so this was treated as a correctness requirement, not a nice-to-have
- Forgot/reset password was built defensively to handle both URL-fragment and PKCE `code`-param recovery link formats, since `detectSessionInUrl: false` means this app must parse the link by hand and there was no way to confirm which format Supabase would actually deliver without a live email round-trip in an environment that can't do one
- Password recovery uses Supabase's email-code (OTP) flow, not a clickable magic link — chosen specifically because this is a mobile-only app with no hosted web page to bounce a browser redirect through, and modern mobile browsers block gesture-less redirects to custom URL schemes (confirmed by the user actually hitting this as a blank page, not theoretical). Any future redirect-based flow in this app (deep links for other purposes are fine — those are opened directly by the OS, not via an intermediate server redirect) should default to a code/OTP approach unless a real hosted redirect page exists
